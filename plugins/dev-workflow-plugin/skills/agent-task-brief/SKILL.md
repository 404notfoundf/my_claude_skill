# Agent Task Brief

## 概览

把"人能看懂但 Agent 容易误解"的需求，改写成可以直接交给 Coding Agent 执行的任务简报。
优先提升规划质量，不靠堆字数；重点是提前写清约束、边界和成功标准。

## 工作流

### 0. 先判断信息是否充足，再决定是直接出简报还是先询问

在开始改写之前，先评估用户给的信息能不能写出一份有效的任务简报。判断分两档：

**信息充足** → 直接进入 Step 1-5，产出结构化简报。

**信息不足**（以下任何一项缺失或过于模糊）→ 先询问，但不要只问问题就停了：

- **目标不明确**：用户说了要做什么，但没说最终要达成什么效果
- **范围不清**：涉及哪些文件/模块/服务不清楚，改完和没改的区别看不出来
- **关键约束缺失**：技术栈、复用要求、禁止事项、测试要求都没有提及
- **没有参照物**：用户描述了一个功能但没给任何现有实现、设计稿或示例

**询问 + 草稿并行**：列出缺失信息让用户补充的同时，基于你的最佳推测给出一份带 `[待确认]` 标记的任务简报草稿。这样用户既有地方可以纠正，也有东西可以直接确认后使用，而不是面对一堆问题无从下手。草稿中的假设部分用 `> **[待确认]**` 块引用标记，与已确认信息明确区分。

不要一次性抛太多问题，优先问对结果影响最大的那一项。

### 0b. 根据输入格式预处理

不同来源的需求需要不同的预处理方式：

- **会议记录/语音转文字** → 先提取行动项，去掉口语化表达和冗余讨论
- **Issue 描述** → 保留原始编号和优先级，提取可编码的规格
- **Slack/聊天记录** → 过滤无关对话，只保留需求相关的部分
- **文章/文档** → 提炼方法论和实现要求，去掉背景铺垫
- **截图/设计稿** → 提取可编码的规格（字段、布局、交互），不臆造实现细节

### 0c. 多任务拆分

如果用户的需求包含多个独立功能（如"加登录+加注册+加权限"），拆分为多份简报，每份聚焦一个可独立执行的任务。在对话中列出所有简报的标题，让用户确认拆分是否合理。

### 0d. 从审查文档提取任务

如果用户提供了审查文档（如 `docs/working_doc/` 下的文件），从中提取待修复条目，每个条目生成独立的任务简报：

- 读取 `docs/working_doc/` 下的文档
- 对每个 P0/P1 条目生成独立简报
- 复用条目中的：文件位置、风险说明、修复建议
- 验收方式从审查文档的"验收建议"段落提取
- 简报标题格式：`[审查文档名] - 条目标题`

### 1. 识别用户真正要改什么

区分"想要结果"与"随口举例的实现方式"，避免把模糊意图直接翻译成错误实现。

### 2. 补齐四类关键信息

- 上下文：相关模块、现有模式、约定文档、依赖边界
- 范围：改哪些，不改哪些
- 约束：技术栈、库选择、风格规范、迁移限制、测试要求
- 验收：什么算完成，怎么验证

### 3. 把隐式假设写成显式指令

如果用户默认 Agent "应该知道"的东西实际没有写出来，就补进任务简报。

### 4. 生成可直接执行的 Markdown 任务简报

让另一个 Coding Agent 拿到后可以先看什么、改什么、不能做什么、最后怎么验。

### 5. 遇到缺失决策时，不要擅自脑补

明确标注假设，或列出必须补充的问题。

### 6. 输出前自检

在输出简报前，检查以下三点：

- **目标是否可观察**：不是"优化性能"，而是"响应时间 < 200ms"或"测试覆盖率达到 80%"
- **约束是否具体**：不是"遵循规范"，而是"参考 auth_handler.go 的模式"或"使用项目已有的 zap 日志库"
- **验收是否可执行**：有具体的测试命令、检查步骤或可观察的行为变化

如果某项不满足，补充具体化后再输出。

自检后给出质量评分（1-5）：
- **1-2**：信息严重不足，需要大量补充
- **3**：基本可用，但有假设需要确认
- **4**：完整可执行，假设合理
- **5**：完美，可直接交给 Agent

评分附在简报末尾，格式：`> 质量评分：4/5`

## 默认结构

除非任务非常小，否则优先使用下面五段结构：

1. 目标
   用一两句话说明最终要达成的结果。
   优先级：P0/P1/P2（可选，用于多任务时排序）

2. 上下文
   写清要先查看的模块、文件、文档、既有实现或依赖关系。

3. 范围与边界
   说明这次改动包含什么、不包含什么、做到什么程度算完成。

4. 约束
   写清必须遵守的架构决策、复用要求、禁止事项、测试与提交流程。

5. 验收方式
   给出可操作的验证方法，包括测试命令、行为检查、边界场景或输出物。

如果只是简单的单文件小改动，可以压缩第 2 段和第 5 段，但不要把边界和约束完全省掉。

简报长度参考：
- 单文件小改动：200-400 字
- 跨模块功能：500-1000 字
- 复杂重构：1000-2000 字
- 原则：够用就行，不靠字数取胜

## 写作规则

- 优先写明确决策，不写笼统目标。
- 默认保持现有架构，除非用户明确要求重构或改设计。
- 区分"已知事实"和"你的推断"。
- 不要把用户没要求的实现方案塞得过满。
- 如果仓库里已有 `CLAUDE.md`、`AGENTS.md`、`README`、设计文档，直接引用具体规则来源，不要泛泛地说"遵循现有规范"。
- 如果用户说 Agent 之前跑偏过，加一个 `避免事项` 小节，明确指出之前的错误方向。示例：

```markdown
## 避免事项
- 不要用 mock 数据替代真实 API 调用
- 不要修改 user_handler.go 以外的文件
- 不要引入新的状态管理库
- 不要重构现有的认证中间件
```

## 输出要求

默认输出可直接复制给 Agent 的 Markdown 任务简报。

**信息不足时的输出**：不要只列问题清单。始终附带一份基于推测的简报草稿，用 `> **[待确认]**` 标记假设部分。格式：

```markdown
## 目标
[你对目标的理解]

> **[待确认]** 用户未明确指定 XXX，这里假设为 YYY。

## 上下文
...

## 范围
...

## 约束
...

## 验收
...
```

这样用户只需要确认或修改假设，而不是从零开始描述需求。

如果用户要的是"模板"，输出：

- 一个通用模板
- 一个基于当前需求填写后的示例

如果用户要的是"把当前需求改写好"，直接给成品，不要先讲理论。

**简报迭代**：用户确认草稿后可能还会要求修改（如"第 3 点改一下""范围再窄一点"）。处理方式：
- 直接修改对应段落，输出更新后的完整简报
- 在修改处用 `<!-- updated -->` 或加粗标记变更内容，方便用户快速定位
- 不要输出"修改前 vs 修改后"的对比，只输出最终版本
