# 文章提炼

来源文章：
`https://mp.weixin.qq.com/s/HbETDyY9nj8HHCYLokhFVQ`

## 核心观点

文章的核心判断是：多数 Coding Agent 失败，问题出在规划阶段，而不是执行阶段。
更具体地说，Agent 跑偏，往往不是模型能力不够，而是任务描述没有提前把上下文、边界、约束和验收标准讲清楚。

## 常见失败信号

出现下面这些表述时，优先触发本 skill：

- “Agent 跑了很久，但方向错了”
- “帮我把这个需求整理成 Claude Code / Cursor 能直接执行的 prompt”
- “把这段模糊需求改成结构化任务说明”
- “根据文章/规范整理一个任务模板”
- “把项目里的规则写进任务描述里”

## 五段式任务开头

重写需求时，优先组织成以下五段：

1. 目标
   说明想达成什么结果，必要时补一句业务目的。

2. 现有上下文
   告诉 Agent 应先看哪些模块、文件、文档、现有实现或系统边界。

3. 范围
   说清楚本次修改包含哪些内容，不要改哪些内容。

4. 约束
   明确技术与流程限制，例如：
   - 复用现有技术栈
   - 优先沿用已有库，不引入新库
   - 不改数据库结构
   - 遵守 `CLAUDE.md` / `AGENTS.md`
   - 不改无关文件

5. 验收
   说明如何判断成功，例如：
   - 运行哪些测试
   - 哪些用户行为必须正确
   - 哪些边界场景必须覆盖
   - 需要产出哪些文件或结果

## 重写启发式

- 把”支持 X”改写成”在什么模块、按什么现有模式支持 X”。
- 把”优化一下”改写成”要解决的具体问题和可观察结果”。
- 把”遵循现有风格”改写成”参考哪个文件、模块或规范文档”。
- 不知道的约束不要编，列成假设或待确认项。
- 简单任务可以压缩模板，但不要丢掉范围和约束。

## 信息不足时的处理原则

当用户需求缺少关键信息时，常见错误是”只列问题清单，等用户回答后再动手”。这会让用户面对一堆问题无从下手。

正确的做法是**询问 + 草稿并行**：

1. 列出你缺失的关键信息（最多 3-5 项，优先级最高的先问）
2. 同时基于你的最佳推测，给出一份完整的简报草稿
3. 草稿中的假设部分用 `> **[待确认]**` 标记
4. 用户只需要说”第 2 项改成 XXX，其他没问题”即可

这样用户的认知负担最低 — 不是从零描述需求，而是确认或修改你的理解。

## 精简模板

```md
## 目标
[一句话说明结果]

## 上下文
- 先查看：[文件 / 模块 / 文档]
- 复用：[现有实现 / 既有模式]

## 范围
- 包含：[这次要做的改动]
- 不包含：[明确不要动的部分]

## 约束
- [架构或工具规则]
- [验证、提交流程或迁移限制]

## 验收
- [测试命令或行为检查]
- [边界场景]
```

## 重写示例

原始需求：

```text
加一个登录功能。
```

改写后：

```md
## 目标
在现有 Web 应用中接入用户登录能力，沿用当前认证体系，不做整体认证重构。

## 上下文
- 先查看 `internal/auth/`、现有 session middleware，以及当前用户模型。
- 遵守仓库中的 `CLAUDE.md` 和项目级认证约定。

## 范围
- 新增登录入口、认证服务逻辑和必要的路由接线。
- 不改注册流程，不替换当前 session 机制。

## 约束
- 优先复用现有认证库和 middleware 链路。
- 除非现有方案无法满足需求，否则不要引入新的 OAuth 库。

## 验收
- 正确账号密码可以成功创建会话。
- 错误账号密码返回现有错误结构。
- 相关认证测试通过。
```
