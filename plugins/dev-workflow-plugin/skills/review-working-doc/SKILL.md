# Review Working Doc

## Overview

用于处理“代码审查/实现优化/问题分析”这类请求，并在条目很多或用户明确要求整理文档时，自动切换到“先写工作文档、再输出结论”的方式，避免对话内容过长且难以维护。

重点识别：

- 正确性风险
- 可维护性问题
- 结构重复
- 性能热点
- 测试缺口
- 文档结构问题

## Workflow

1. 判断用户请求是否属于审查/优化/分析类型，并确认审查范围：
   - 用户指定了具体文件或函数 → 直接进入
   - 用户说”这里/这个地方” → 从最近讨论的文件或当前上下文中定位
   - 用户说”审查一下代码”但没指定范围 → 询问或从最近修改的文件（git log）入手
   - 如果审查范围超过 5 个文件，按优先级分批处理：先处理最近修改的文件，再处理核心业务逻辑，最后处理基础设施层

2. 读取目标代码，同时判断内容类型和模板选择：
   - **问题/整改台账**：适用于安全风险、漏洞、行为错误、权限缺陷、需跟踪修复的事项
   - **优化建议清单**：适用于结构改进、可维护性建议、文档整理、非缺陷类改造点
   - **局部代码分析**：适用于单文件/单函数的实现分析，风险与优化点并存
   - 如果信号混合，优先展示混合列表，让用户确认模板类型，而不是自动选择
   - 在 `docs/working_doc/` 下查找是否有同主题的历史文档，有则在其基础上更新

3. 对技术条目分级：
   - `P0`：错误、风险、行为不一致、可能导致故障的点
   - `P1`：结构和职责问题，后续维护成本高
   - `P2`：风格、命名、轻度重复、小型重构机会

4. 如果最终条目不超过 3 条，且用户没有明确要求落文档：
   - 直接在对话中给出结论。

5. 如果最终条目超过 3 条，或者用户明确要求整理工作文档：
   - 在 `docs/working_doc/` 下生成 Markdown 工作文档。
   - 文件名格式：`YYYYMMDD_<topic>_optimization.md`，`<topic>` 用简短英文蛇形命名。
   - 若同日同主题文档已存在，更新现有文件而不是创建新文件。

6. 生成工作文档后：
   - 将完整条目写入该文档。
   - 在对话里列出文档路径和关键发现，请用户确认是否有遗漏或需要调整的条目。

7. 如果用户在文档生成后要求”改一下””修复””实施”：
   - 基于文档中的修复建议，逐项执行代码修改。
   - 每完成一项，更新原文档中对应条目的 checkbox 状态（`[ ]` → `[x]`）。
   - 每完成一项修改后，运行相关的测试或编译验证；如果修改引入了新问题，回滚该条目并标注为”修复失败”。
   - 在对话里汇报已完成的项和剩余项。
   - 始终更新原始工作文档，不创建新文档。

8. 如果用户要求审查多个文件或整个目录：
   - 按 bounded context 或模块拆分，每个逻辑单元生成一个独立文档。
   - 文件名格式：`YYYYMMDD_<module>_review.md`。
   - 在对话里先给出跨模块的总览摘要，再列出各文档路径。
   - 如果条目之间有关联（如同一个问题影响多个模块），在各文档中交叉引用。

## Writing Rules

- 工作文档统一放在当前项目的 `docs/working_doc/` 目录下。
- 不要把这类长条目分散到多个临时文件里；同一轮任务优先维护一个工作文档。
- 文件名必须带日期，且应包含简短主题，避免只写成无意义名称。
- 日期使用当天本地日期，格式优先使用 `YYYYMMDD`。
- 每条优化项都要说明”为什么值得改”。
- 优先引用具体文件和函数。
- 如果某项只是想法或推测，要明确标注。
- 如果用户明确是在整理已有 Markdown 文档，优先更新原文档，而不是新建第二份。
- 对于问题/整改台账：
  - 总览表作为唯一状态来源
  - `修复` 列只放 checkbox：`[ ]` 未完成，`[x]` 已完成
  - 总览表列优先使用：`ID | 级别 | 问题 | 修复 | 风险类型 | 备注`
  - `风险类型` 列用于快速分类
  - `备注` 列用于未修复原因、阻塞项、接受风险、排期说明
  - 详情中不要重复写状态
- 对于优化建议清单：
  - 总览表列优先使用：`ID | 优先级 | 优化点 | 建议类型 | 预期收益 | 备注`
  - 默认不使用 checkbox，除非用户明确要跟踪”是否采纳/是否完成”
- 对于局部代码分析：
  - 总览表列优先使用：`ID | 类型 | 标题 | 优先级 | 处理建议 | 备注`
  - 默认不使用 checkbox

## Document Structure

根据内容类型，生成到 `docs/working_doc/` 的 Markdown 文档默认优先使用以下结构：

- 问题/整改台账：
  1. 文档标题
  2. 简短说明
  3. 总览
  4. 详情
  5. 推荐实施顺序
  6. 验收建议
- 优化建议清单：
  1. 文档标题
  2. 简短说明
  3. 总览
  4. 详情
  5. 推荐采纳顺序
- 局部代码分析：
  1. 文档标题
  2. 简短说明
  3. 总览
  4. 详情
  5. 建议处理顺序

参考模板：

- 问题/整改台账：`references/issue-template.md`
- 优化建议清单：`references/optimization-template.md`
- 局部代码分析：`references/code-analysis-template.md`
